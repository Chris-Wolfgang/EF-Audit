Remarks

Use this when your DbContext already inherits from a third-party base (such as IdentityDbContext<TUser>, a multi-tenant base, or an internal enterprise base) and cannot also inherit from AuditingDbContext. Register via: 

services.AddEfCoreAuditing<MyUserProvider>();
services.AddDbContext<AppDbContext>((sp, opts) => opts
    .UseSqlServer(connStr)
    .UseAuditing(sp));

Cancellation. If the CancellationToken is signalled while SavingChangesAsync is opening the audit transaction, the interceptor catches the resulting OperationCanceledException, disposes the owned transaction, and re-raises — no leak. Cancellation that fires later (during EF's own SQL execution between SavingChangesAsync and SavedChangesAsync) is handled via SaveChangesFailedAsync on EF Core 6 / pre-7-cancel-callback versions; on EF Core 7+ that routes through SaveChangesCanceledAsync instead, which this interceptor doesn't yet hook because that method isn't on the EF Core 6 ISaveChangesInterceptor surface. In that narrow window the owned transaction is disposed when the DbContext is. Inherit from AuditingDbContext for full cancellation robustness — it owns the SaveChangesAsync override and can clean up on every exit path.

acceptAllChangesOnSuccess caveat. The interceptor does not support SaveChanges(acceptAllChangesOnSuccess: false): when the caller passes false, EF leaves user entries in their dirty state and the interceptor's audit-pass save would re-emit them. The interceptor detects this case at runtime and throws InvalidOperationException pointing at AuditingDbContext, which threads the value through both passes correctly.

Retry-strategy caveat. EF Core's retrying execution strategies (e.g. SqlServerRetryingExecutionStrategy from EnableRetryOnFailure) refuse user-initiated transactions opened outside a strategy delegate. Important: strategy.ExecuteAsync(...) by itself does NOT open a transaction — Database.CurrentTransaction remains null inside it, and this interceptor would still try to open its own owned transaction (then throw via EnsureNonRetryingStrategy). The supported integration is either:

  • Call strategy.ExecuteInTransactionAsync(...) so the strategy opens a transaction the interceptor can enlist in, OR
  • Inside your ExecuteAsync delegate, call context.Database.BeginTransactionAsync(...) yourself before SaveChangesAsync, OR
  • Inherit from AuditingDbContext instead — it composes with retrying strategies correctly out of the box.

Rollback / ChangeTracker divergence. If the audit-pass SaveChanges throws after the user-pass SaveChanges has already accepted the user entries (entries are now Unchanged), rolling back the owned transaction puts the database back to the pre-save state but the in-memory ChangeTracker still reflects the accepted state. The interceptor calls Clear() on the rollback path so a stale-state reuse fails loudly the next save — but the safer pattern is to dispose the DbContext after a save failure and create a new one.
