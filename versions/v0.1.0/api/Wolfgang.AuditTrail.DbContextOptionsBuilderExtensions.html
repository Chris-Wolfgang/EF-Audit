UseAuditing(DbContextOptionsBuilder, IServiceProvider)

Registers the audit save-changes interceptor on this DbContextOptionsBuilder. Use this for applications whose DbContext already inherits from a third-party base (e.g. IdentityDbContext<TUser>) and therefore cannot also inherit from AuditingDbContext.

public static DbContextOptionsBuilder UseAuditing(this DbContextOptionsBuilder optionsBuilder, IServiceProvider serviceProvider)

Parameters

optionsBuilder DbContextOptionsBuilder

The EF Core options builder.

serviceProvider IServiceProvider

The DI service provider; must already contain AuditOptions and IAuditUserProvider (register them via services.AddEfCoreAuditing<TUserProvider>()).

Returns

DbContextOptionsBuilder

Remarks

You must also call modelBuilder.ApplyAuditing(options) from your DbContext.OnModelCreating override. Unlike AuditingDbContext (which applies the audit entity mappings itself in its OnModelCreating), the interceptor cannot reach into the consumer's model — the EF Core model is sealed by the time the interceptor first runs. Without the ApplyAuditing call, the first save with audit data will throw an "entity type was not found in the model" exception when the interceptor tries to Add the AuditHeader / AuditDetail entities.

Minimal correct wiring:

services.AddEfCoreAuditing<MyUserProvider>();
services.AddDbContext<AppDbContext>((sp, opts) => opts
    .UseSqlServer(connStr)
    .UseAuditing(sp));

public class AppDbContext : IdentityDbContext<AppUser>
{
private readonly AuditOptions _auditOptions;
public AppDbContext(DbContextOptions options, AuditOptions auditOptions)
: base(options) => _auditOptions = auditOptions;


protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    base.OnModelCreating(modelBuilder);
    modelBuilder.ApplyAuditing(_auditOptions);  // <-- required
}



}
